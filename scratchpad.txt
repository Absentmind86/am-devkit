--- CODE QUALITY / HOUSEKEEPING ---

Catalog adoption is ~80-85% complete (not fully unified).
The split between pre-catalog and catalog-driven installs is currently implicit — needs
to be formalized as a first-class architectural boundary (not just accepted as debt).

GENUINELY CANNOT MOVE TO CATALOG (true bootstrap prerequisites):
  - Git, Python 3 — required before the Python runtime + catalog system can run at all
  - Scoop — required before any Scoop-based tool can be installed
  - Git LFS — must follow Git, pre-catalog ordering matters
  THESE STAY in infrastructure.py / languages.py. Document clearly in AGENTS.md.

SHOULD MOVE TO CATALOG (no bootstrap reason, just installed early by habit):
  - Oh My Posh — cosmetic shell enhancement, zero bootstrap dependency
  - Tailscale — network tool, no ordering requirement
  - uv — Python package manager, catalog-installable after Python exists
  - GitHub CLI — useful early but not a prerequisite for anything
  - Windows Terminal, PowerShell 7 — no strict ordering dependency
  ACTION: Move these to a new "foundation" layer in WINGET_CATALOG with profiles=None.
  This makes them excludable via --exclude-catalog-tool like VS Code/Cursor already are.

Enterprise approach: formalize the boundary in code + docs, not just comment it.
  - Add a BOOTSTRAP_TOOLS block in infrastructure.py with a clear docstring explaining
    WHY these bypass catalog (ordering constraint, not convenience)
  - Move everything without a genuine bootstrap reason into catalog
  - Document the two-tier pattern explicitly in AGENTS.md as an architecture decision

languages.py is the most inconsistent module:
- Python + uv: direct winget (bypass catalog)
- Node: manual print instruction only (no automation)
- Rust: custom helper + two different skip reasons ("user-opt-out" vs "profile-gate")
- Final install_catalog_layer("languages") call looks like a partial migration leftover

_ensure_repo_on_sys_path() confirmed duplicated in: installer.py, gui.py, system_scan.py,

--- CRITICAL GAPS (PRE-RELEASE BLOCKERS) ---

WSL restart handling — no resume logic:
- DISM runs with /norestart (correct) and exit 3010 is noted (pwsh_util.py:358-360)
- BUT: no prompt, no pause, no resume-after-reboot orchestration
- On fresh machines WSL enable + Docker install may silently fail or require manual re-run
- Should warn user explicitly in pre-install summary that a reboot + re-run may be required
  when WSL is being enabled for the first time

VM detection gap:
- system_scan.py collects Win32_ComputerSystem.Manufacturer + Model (lines 104-105, 287)
  which often contains "VMware", "VirtualBox", "Microsoft Corporation" etc.
- This data is NEVER used to set any is_vm flag in system-profile.json or InstallContext
- gpu_detect.py has no VM/hypervisor detection
- On VMs: GPU detection usually falls back to CPU safely, but no warning is shown
- Recommendation: Add is_vm detection in system_scan.py using existing Manufacturer/Model
  data; surface warning in GUI/summary when ai-ml + --install-ml-wheels selected on a VM

--- TRANSPARENCY AUDIT SUMMARY ---

Overall verdict: Very Good (8.5/10). Strengths:
- Excellent pre-install summary with estimates + confirmation prompt
- Per-layer progress output with fault isolation (_safe_layer)
- Comprehensive HTML report + manifest with every tool timestamped
- Explicit Y/N for Flet downgrade at bootstrap

Gaps to address:
1. WinUtil: user sees preset name only, not the actual tweak list — show tweak list in
   pre-install summary when sanitation is enabled
2. No per-layer "about to install: X, Y, Z" preview before each layer runs
3. No automatic post-install validation step (addressed by Validator feature above)

- install_catalog.py + catalog_install.py: strong centralized system, clean design

- Manifest schema am-devkit-manifest-1.1: consistent across all layers, no drift found
- bootstrap/install.ps1 + fresh.ps1: correct handoff to Python, clear error messages
- All profile TOMLs consistent in structure (even if content is stubs)
- Third-party notices and license: accurate
